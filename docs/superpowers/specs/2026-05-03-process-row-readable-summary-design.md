# 进程行可读摘要设计规格

## 目标

降低进程列表的阅读成本，让用户第一眼看到“哪个项目在跑什么”，而不是先看到很长的可执行路径或 `node_modules` 路径。

## 非目标

- 不改变进程扫描、分类、过滤或停止逻辑。
- 不移除完整命令行信息。
- 不新增设置面板或自定义展示模式。
- 不改变来源、项目分组和搜索行为。

## 展示设计

每个进程行使用两层信息：

1. 主标题：`项目名 · 服务类型`
   - 示例：`kill-tool · 开发服务`
   - 示例：`my-blog · MCP`
   - 示例：`未识别项目 · 脚本`
2. 辅助行：`命令动作 · PID · 来源 · 运行时间`
   - 示例：`npm run dev · PID 12345 · Codex · 12m`
   - 示例：`next dev · PID 39869 · VS Code · 2h`
   - 示例：`playwright-mcp · PID 48123 · Codex · 8m`

端口继续保持在主标题右侧，安全标签和类型标签继续保留。完整命令行降级为悬停提示或详情文本，不再作为行首主要内容。

## 命令动作提取

展示层从 `commandLine` 中提取短命令动作：

- 优先识别常见开发命令：`npm run dev`、`pnpm dev`、`yarn dev`、`bun dev`、`next dev`、`vite`、`astro dev`、`webpack`。
- 识别 MCP 相关命令：`mcp-server-*`、`playwright-mcp`、`node_repl`。
- 识别数据库和基础设施命令：`postgres`、`redis-server`、`docker-proxy`。
- 识别不到时回退到 `executableName`。
- 仍然保留完整 `commandLine`，用于搜索和悬停查看。

## 交互细节

- 列表默认显示短摘要，避免长路径撑满行宽。
- 悬停行内容时可以通过 tooltip 查看完整命令。
- 搜索仍覆盖项目名、来源、类型、安全状态、可执行名、完整命令、PID 和端口。
- 保护、安全、谨慎标签保持原含义。

## 测试策略

- 添加或更新展示摘要的单元测试，覆盖常见命令、MCP 命令和回退逻辑。
- 运行 Swift 测试，确保进程扫描和分类行为不变。
- 手动检查 popover 行内容是否从路径优先变为项目和服务优先。

## 验收标准

- 进程行主标题不再直接显示完整 `commandLine`。
- 用户能从主标题快速判断项目和进程类型。
- 用户仍能查看完整命令行。
- 搜索完整路径或命令仍能匹配对应进程。
- 停止、强制结束、选择子进程等操作行为不变。
